# BlackSand Executive Leasing Dashboard

**Version 1.0.0** — production-ready (see `CHANGELOG.md`, `RELEASE_CHECKLIST.md`, and
`DEPLOYMENT_CHECKLIST.md`).

## Project overview

The **BlackSand executive real-estate leasing dashboard** is a single-screen, always-on
"situation board" for commercial leasing performance (occupancy, leased area, KPIs,
leasing velocity, trends, top tenants, per-building occupancy, and a searchable tenant
directory), designed for **large office TVs / 4K displays**.

- Built with **vanilla HTML, CSS, and JavaScript** — no framework, no build step.
- Uses **Three.js** (occupancy centrepiece, particle-wave background, tenant medallions)
  and **Chart.js** (mini bar charts), both from a CDN.
- Runs two ways: **standalone** (open the HTML file) or **served locally** via
  **Node.js + Express**.
- **Not yet connected to Monday.com** — data is currently embedded in the frontend.

The standalone `Project Dashboard.html` is the **reference implementation**. Express is
an *optional* host that serves that exact same file; it never modifies the frontend.

## Database (Phase 1 — SQLite foundation)

The backend uses an **embedded SQLite database** via
[`better-sqlite3`](https://github.com/WiseLibraries/better-sqlite3). SQLite is a
file — there is **no separate database server to install or run**. `better-sqlite3`
is installed automatically by `npm install`.

- **Default location:** `data/dashboard.db` (created on first migration).
- **Configurable** via the `SQLITE_DB_PATH` environment variable. Relative paths
  resolve from the **project root** (not the terminal working directory); an
  absolute path is honoured as-is; the parent directory is created if missing.
- **Generated files are gitignored:** `data/*.db`, `data/*.db-wal`, `data/*.db-shm`
  (WAL sidecars) and `data/backups/` are never committed. `data/.gitkeep` keeps the
  directory in the repo. The migration/schema **source code** is committed.
- **One Node process** is assumed (a single writer). Reliability pragmas applied on
  every open: `foreign_keys = ON`, `journal_mode = WAL`, `synchronous = NORMAL`,
  `busy_timeout = 5000`.

### Database commands

```bash
npm run db:migrate   # create the DB if missing + apply pending migrations, then validate
npm run db:check     # validate schema + pragmas WITHOUT changing anything (read-only)
```

`db:migrate` is the only command that creates or upgrades the schema; `db:check`
only reports. Re-running `db:migrate` is safe (already-applied migrations are
skipped — no duplicate tables or migration rows).

### Typical first run

```bash
npm install       # installs express + better-sqlite3
npm run db:migrate # creates data/dashboard.db and the empty schema
npm run db:check   # confirms the database is usable
npm run check      # static project integrity check
npm start          # starts the server (also migrates on startup)
```

The server **initialises and migrates the database before it starts accepting
requests**, and closes SQLite cleanly on shutdown. Readiness is reported at
`GET /ready` (see below).

## Database seed (Phase 2 — bootstrap data)

Phase 2 migrates the dashboard's **current embedded prototype data** into SQLite via
a safe, repeatable seed. This proves the relational schema can represent the current
dashboard before the frontend starts reading from an API (Phase 3).

- The seed data is **bootstrap / demo data**, transcribed from the `projects[]` array
  in `Project Dashboard.html`. It is written with **`source = 'seed'`** and is **not**
  Monday live data and **not** verified production truth.
- The reviewed seed lives in `server/seed/current-dashboard-data.js` (data only).
- The seed is **repeatable and atomic** — re-running never duplicates rows and yields
  the same canonical `dataVersion` (a SHA-256 over the normalized business data).
- **The frontend still uses its embedded data in this phase** — seeding the database
  changes nothing the dashboard displays. The API will read SQLite in the next phase.
- The database stays **local and gitignored**; no project/tenant data is committed.

### Seed commands

```bash
npm run db:migrate        # ensure schema is current (adds v2: leases.source_record_key)
npm run db:seed:inspect   # load + validate the seed WITHOUT touching the database
npm run db:seed           # atomically seed SQLite (repeatable, source='seed')
npm run db:seed:check     # read-only: verify DB matches the seed module + integrity
npm run db:seed:compare   # reproduce frontend metrics from the seed + literal spot-checks
npm run db:check          # schema/pragma validation
npm run check             # static project integrity check
npm run test:seed         # unit + DB tests for the seed pipeline
```

### Expected warnings (known prototype inconsistencies — NOT bugs)

The seed intentionally **preserves** the current prototype's quirks rather than
"fixing" them (they resolve at Monday.com integration):

- **Town Center** headline occupancy uses an **explicit leased percentage** (retail
  40%, office 69%), while its tenant list is a **separate** dataset. The office
  category's explicit leased area (~6,301 m²) does **not** equal the sum of its lease
  rows (~5,272 m²) — reported as a `WARNING`, preserved deliberately.
- **56 Town Center leases carry prototype-derived mock dates** (the frontend has no
  real dates there); these are deterministic in the seed, not real lease dates.
- **Duplicate tenant names** (Malath, Al Tharwah Co., LABOCCA, …) are **separate
  leases** and are kept as separate rows; tenant count = **lease-row** count.

These warnings are printed by `db:seed` / `db:seed:inspect` and do **not** block seeding.

### Rebuilding the development database safely

There is intentionally **no destructive auto-reset command**. To rebuild the local
dev DB by hand:

1. Stop the server.
2. Confirm the DB is development-only (`data/dashboard.db`, gitignored).
3. Remove the ignored dev DB files manually: `data/dashboard.db`, `data/dashboard.db-wal`,
   `data/dashboard.db-shm`.
4. `npm run db:migrate`
5. `npm run db:seed`
6. `npm run db:seed:check` (and `npm run db:seed:compare`).

> **Phase 2 scope.** No Monday.com connection, no dashboard API, no historical
> snapshots, and no frontend changes. `performanceSummary` is intentionally **not**
> seeded (it is global prototype data that snapshot-derived trends will replace later).

## Dashboard data API (Phase 3) & frontend live data (Phase 4)

The Express host now exposes a **read-only, SQLite-backed JSON API**, and the
**server-hosted** dashboard reads its live data from it. The standalone HTML file
remains a self-contained **demo**.

### API endpoints (read-only, same-origin, no auth, no CORS)

```
GET /api/dashboard      → { data: { projects: [...] }, meta: {...} }
GET /api/sync/status    → seed/sync provenance (last successful sync, dataVersion, …)
```

`/api/dashboard` returns projects in the exact shape the dashboard renderer expects
(`project`, `slug`, `address`, `retail`, `office`, `buildings`) plus a canonical
`metrics` block, and a `meta` block:

| meta field | meaning |
|---|---|
| `checkedAt` | server time this response was produced (a fresh SQLite read) — drives **Last Checked** |
| `dataVersion` | hash of the current data; the frontend re-renders **only** when this changes |
| `lastSuccessfulSync` | when the data was last (re)loaded into SQLite (a real load, not a browser refresh) |
| `lastDataChange` | when the data content last actually changed |
| `source` | always `sqlite` in this phase |

There are **no write routes** (no POST/PUT/PATCH/DELETE) and the API never leaks the
database path or any secret. When the database has no data it returns a controlled
`503` (never fabricated projects).

### Two frontend modes

- **Server mode** (opened over `http://` via Express) — the dashboard fetches
  `/api/dashboard`, renders the SQLite-backed projects, and **polls every five
  minutes**. This is the path toward production.
- **Standalone demo mode** (opened via `file://`, or `?mode=demo`) — uses the embedded
  prototype data, makes **no** network request, and does **not** poll. For offline
  demos and frontend-only testing.

The mode is chosen explicitly by protocol (`file:` → demo, otherwise live). An API
failure in server mode **never** silently switches to demo data — it shows a controlled
error/degraded state instead.

### Live-mode behaviour (what to expect on a TV)

- **Last Checked** shows the API's `checkedAt` — the time of the most recent
  *successful* read (normally 0–5 minutes old). A **manual browser refresh** performs a
  new database read, so it updates Last Checked; it does **not** change `lastDataChange`.
- A poll where `dataVersion` is unchanged updates only Last Checked — it does **not**
  re-render, recreate charts, or reinitialise Three.js.
- A poll where `dataVersion` changed re-renders in place, preserving the selected
  project.
- A **failed** poll keeps the last good view (a subtle "degraded" state) and does
  **not** advance Last Checked; it retries and recovers automatically.
- An **initial** failure (never loaded) shows a restrained "Data unavailable" state and
  retries with capped backoff — it does not show demo values as if they were live.
- URL project selection (`?project=business-address` / `?project=town-center`) is
  preserved; project-tab switching is local and triggers **no** API request.

### Commands

```bash
npm run api:check      # boot the app in-process and validate the API contract
npm run test:api       # API route tests (temporary seeded DB)
npm run test:frontend  # frontend client logic tests (mode/poll/dataVersion/failure)
npm run verify         # Phase 5: run ALL non-destructive checks + test suites in order
```

`npm run verify` is the single command that proves the whole SQLite-backed system is
healthy (schema, seed, metrics, API contract, static invariants, and all test suites).
It never seeds or mutates business data and stops at the first failing stage. Run it
after a migrated + seeded database exists (a normal `npm start` auto-seeds an empty DB).

Server mode:

```bash
npm start
# http://localhost:3000            (live, SQLite-backed)
# http://localhost:3000/?project=town-center
```

On startup the server **auto-seeds the bootstrap data if the database is empty**, so a
plain `npm start` serves live data immediately (no separate `npm run db:seed` needed for
a first run). If the database has no data for any reason, `/api/dashboard` returns a
controlled `503` and the dashboard shows a restrained "Data unavailable" state rather
than embedded values.

Standalone demo mode: **double-click `Project Dashboard.html`** (or open it via
`file://`). After Monday.com integration, live data will still require server mode.

> **Phase 4 scope.** Monday.com is still **not** connected; there is no scheduled sync
> and no stale-while-revalidate against Monday yet. The five-minute poll reads the
> local SQLite-backed API only. A dev/test poll override exists
> (`window.__DASHBOARD_TEST_POLL_INTERVAL_MS__`, or `?pollMs=` on localhost only,
> clamped to ≥1s); production stays at five minutes.

> **Phase 6 scope (Monday foundation, OFFLINE).** The complete Monday integration
> framework now exists under `server/monday/` (client, GraphQL builder, column-type
> adapters, mapper, validator, transformer, difference engine, atomic persistence, sync
> pipeline) but makes **no network request** and needs no credentials yet: sync is
> disabled by default and the client's transport is disabled. Board/column mapping lives
> in `config/monday-mapping.json` (gitignored; copy `config/monday-mapping.example.json`).
> `/ready` reports a Monday health block of booleans only (no secrets). `npm run
> test:monday` covers it offline. Phase 7 will paste the token + IDs, enable sync, and
> inject a real transport — the dashboard, API, and SQLite schema do not change.

> **Phase 7 — Monday.com enabled (live).** The two boards are connected: Town Center
> (`5091991928`) and Business Address (`5094617763`). Requirements: **Node 20.6+** (Node
> 24 recommended); `npm install`; put your token in `.env` as **`MONDAY_API_KEY`**
> (`.env` is auto-loaded by `server/config/load-env.js`; `MONDAY_API_TOKEN` still works
> as a deprecated alias). Board + column IDs live in **`config/monday-mapping.json`**
> (gitignored; copy `config/monday-mapping.example.json`) — not in env vars. Safe offline
> mode is the default (`MONDAY_SYNC_ENABLED` unset/false → no Monday call). Commands
> (all read-only against Monday except the gated write):
>
> - `npm run monday:inspect-board -- <board-id>` — board name, groups, column IDs/types.
> - `npm run monday:inspect-sample -- <board-id> [n]` — tiny redacted item sample.
> - `npm run monday:mapping:check` — validate the mapping (rejects placeholders).
> - `npm run monday:mapping:validate-live` — confirm mapped column IDs exist on the boards.
> - `MONDAY_DRYRUN_LIVE=true npm run monday:dry-run` — live read-only preview, zero writes.
> - **Real sync (gated, CLI-only):** `MONDAY_SYNC_ENABLED=true MONDAY_DRY_RUN=false npm run monday:sync -- --confirm`
>   — reads Monday, writes SQLite atomically, cuts each project over to the Monday source
>   (seed rows preserved for fallback). The server never auto-syncs.
>
> `GET /api/sync/status` reports each project's `currentSource` (`seed`/`monday`) and last
> sync. **Never commit `.env`.** A personal Monday token can access only the data that
> Monday user is permitted to see. Troubleshooting: missing token → `monday:ready`
> fails "token not configured"; invalid token → auth error; wrong board ID / no access →
> `inspect-board` errors; missing/invalid mapping → `mapping:check` errors; a
> validation/safety rejection preserves the previous data source unchanged.

> **Pre-Phase-7 hardening.** Source ownership is now explicit: each project has a
> `current_data_source` (`seed` by default) and the dashboard reads leases only from that
> source, so seed and Monday data never count together. The first successful Monday sync
> atomically cuts a project over to `monday` while **preserving** seed rows for fallback.
> Safety guards reject empty boards, record-count collapses, incomplete fetches and
> missing required boards (last-known-good preserved). Total GLA must be explicitly
> configured (never defaulted to 0); lease `is_active` is driven by a canonical status
> (unknown status is rejected, never active); buildings stay manual/seed in v1. Read-only
> tooling: `npm run monday:mapping:check[:draft]`, `monday:inspect-board`,
> `monday:inspect-sample`, `monday:dry-run`, `monday:mapping:validate-live`,
> `monday:ready` (production gate). `npm run monday:sync` exists but **refuses production
> writes until Phase 7**. `npm run verify` runs everything offline (no token, no network).

## Health and readiness endpoints

- `GET /health` — **liveness**: the Node process is alive. Stays `200` even while data is
  being configured (it never depends on the DB/Monday for its status). It now also reports
  operational info for monitoring: `status`, `service`, `version`, `environment`, `uptime`
  (seconds), `database` (`ready`/`unavailable`), `scheduler` (`running`/`idle`/`disabled`),
  and `timestamp`. No secret, path, or stack is exposed.
- `GET /ready` — **readiness**: the database is open with a current, valid schema.
  Returns `200` `{ status: "ready", database: "ready", schemaVersion, … }` (the
  current schema version) when ready, or `503` when not. It intentionally exposes
  **no** database path, table names, pragma details, or environment values.

## Production deployment (Phase 9.4A)

For a permanent office-server deployment (as opposed to `npm start` in a terminal), run
the server under **PM2** so it restarts on crash and after reboot, with validated
configuration and centralized logs. **No business logic changed** — this is operational
hardening only. The full step-by-step is in **`DEPLOYMENT_CHECKLIST.md`**.

### Production configuration & validation

- Set `NODE_ENV=production` in `.env`. In production, configuration is **validated at
  startup**; an invalid value prints a clear, secret-free error and the server **exits
  non-zero** (it never serves misconfigured). In development the same issues are warnings.
- Recognized variables (see `.env.example`): `NODE_ENV`, `PORT`, `HOST`, `LOG_LEVEL`,
  `SQLITE_DB_PATH` (alias `DATABASE_PATH`), `HISTORY_SNAPSHOT_TIME` (alias
  `SNAPSHOT_SCHEDULE`), `HISTORY_TIMEZONE` (alias `TIMEZONE`), `MONDAY_API_KEY` (alias
  `MONDAY_API_TOKEN`), `MONDAY_SYNC_ENABLED`. Monday **board IDs** live in
  `config/monday-mapping.json`, not in an env var.
- The config layer lives in `config/` (`index.js`, `production.js`, `development.js`,
  `validation.js`). It is an additive facade over the existing env handling.

### Logging

- Centralized logger (`server/logger.js`): levels `error|warn|info|debug` gated by
  `LOG_LEVEL`. Each entry has a timestamp, level, source, message, and optional context;
  **secrets are redacted** by key. Output goes to the console (captured by PM2) **and** to
  `logs/dashboard.log` (one JSON line per entry). `logs/` is gitignored.

### PM2

```bash
npm install -g pm2
pm2 start ecosystem.config.js --env production   # 1 instance (SQLite is single-writer)
pm2 save                                          # remember the process list
pm2 startup                                       # print the reboot command; run it once (admin)
pm2 status | pm2 logs blacksand-dashboard
```

`ecosystem.config.js` pins **`instances: 1`, `exec_mode: fork`** (never cluster — exactly
one process may own the SQLite file), `autorestart`, `max_memory_restart: 512M`, and a
`kill_timeout` generous enough for graceful shutdown. Graceful shutdown (SIGINT/SIGTERM)
stops the scheduler, waits for any active snapshot, then closes SQLite.

### Backup & restore

```bash
npm run backup                       # consistent online copy → data/backups/dashboard-<UTC>.db
BACKUP_KEEP=14 npm run backup        # keep only the newest 14 backups
npm run restore <backup-file>            # preview (no changes)
# STOP the server first, then:
npm run restore <backup-file> --confirm  # validates the backup, safety-copies the current DB, restores
```

Backups are validated (`integrity_check`) on creation; restore validates the backup
**before** overwriting and safety-copies the current DB first. The database lives at
`SQLITE_DB_PATH` (default `data/dashboard.db`); `data/*.db*` and `data/backups/` are
gitignored and must be backed up out-of-band as part of server backups.

### Production scripts

```bash
npm start     # run the server (validates config first)
npm run dev   # dev with --watch
npm test      # full offline test suite (seed/api/frontend/production/monday/history)
npm run lint  # dependency-free syntax gate over all project .js
npm run backup / npm run restore   # see above
npm run verify# consolidated non-destructive checks + all test suites
```

### Environment variables

All are optional (sensible defaults apply); set them in `.env` (never commit it). In
**production** an invalid value aborts startup with a clear message.

| Variable | Default | Purpose |
|---|---|---|
| `NODE_ENV` | `development` | `production` makes config validation strict (fail-closed). |
| `PORT` | `3000` | HTTP port. |
| `HOST` | `0.0.0.0` | Bind address (LAN-reachable for TVs). |
| `LOG_LEVEL` | `info` (prod) / `debug` (dev) | `error` \| `warn` \| `info` \| `debug`. |
| `SQLITE_DB_PATH` | `data/dashboard.db` | DB file (relative → project root). Alias: `DATABASE_PATH`. |
| `HISTORY_SNAPSHOT_TIME` | `02:00` | Daily snapshot time (Riyadh). Alias: `SNAPSHOT_SCHEDULE`. |
| `HISTORY_TIMEZONE` | `Asia/Riyadh` | Scheduler timezone (must be Asia/Riyadh). Alias: `TIMEZONE`. |
| `HISTORY_AUTOMATION_ENABLED` | `true` | Daily snapshot scheduler on/off. |
| `MONDAY_API_KEY` | — | Monday token (secret; env-only, never logged). Alias: `MONDAY_API_TOKEN`. |
| `MONDAY_SYNC_ENABLED` | `false` | Enables the gated CLI sync (the server never auto-syncs). |

Monday **board / column IDs** live in `config/monday-mapping.json` (gitignored), **not** in
env vars. See `.env.example` for the full annotated list.

## Finding the Windows IP

Open a terminal and run:

```bash
ipconfig
```

The address the TVs need is the **IPv4 Address** under your active **Ethernet** or
**Wi-Fi** adapter (e.g. `192.168.1.50`). Ignore virtual adapters (VPN, WSL, Hyper-V,
loopback). The Express server also **prints the detected network URLs when it starts**,
so `npm start` is the quickest way to confirm the right address.

## Windows Firewall

The first time you run `npm start`, Windows may pop up a **Windows Defender Firewall**
prompt asking whether **Node.js** may communicate on the network.

- For an office LAN, **allow "Private networks"** (unless company IT says otherwise).
- Leave "Public networks" unticked.
- **Do not disable the firewall.** If a TV cannot reach the dashboard later, re-check this
  Node.js rule in *Windows Defender Firewall → Allow an app through firewall*.

## Stable IP

TVs point at the host PC by IP, so that IP should not change. Have the host PC receive
either:

- a **DHCP reservation** on the router (bind the PC's MAC to a fixed address), **or**
- a **static IP** configured by company IT.

Otherwise a DHCP lease change will silently break every TV's URL.

## Automatic startup

Two **manual** options (nothing is installed or changed on Windows automatically):

**Desired boot sequence:** (1) start the Node server → (2) wait a few seconds → (3) open
Edge in kiosk mode. The example script `scripts/start-dashboard.bat` does exactly this;
edit the `DASH_HOST` / `DASH_PORT` / `DASH_PROJECT` (and `EDGE` path if needed) at the top
first.

**Option 1 — Startup folder**
1. Press `Win+R`, type `shell:startup`, press Enter.
2. Place a **shortcut** to `scripts/start-dashboard.bat` in that folder.
3. It runs at user log-on.

**Option 2 — Task Scheduler**
1. Open **Task Scheduler → Create Task**.
2. Trigger: **At log on** (optionally delay 10–30 s).
3. Action: **Start a program** → point to `scripts\start-dashboard.bat`, and set *Start in*
   to the project folder.
4. Tick **Run only when user is logged on** (a kiosk browser needs a desktop session).

Do **not** add PM2, NSSM, or a Windows service.

## Updating the dashboard

1. **Stop the server** (Ctrl+C in the server window; close kiosk browsers if needed).
2. **Create a Git checkpoint** — `git add -A && git commit -m "checkpoint before update"`.
3. **Apply the updates** (edit `Project Dashboard.html` / assets).
4. **`npm install`** — only if dependencies changed.
5. **`npm run check`** — confirm integrity.
6. **`npm start`** — relaunch the server.
7. **Refresh or restart the kiosk browsers** (HTML is served `no-cache`, so a refresh
   picks up changes).

## Troubleshooting

- **Port 3000 already in use** — another process holds it. Stop it, or run with a
  different port: `set PORT=3001 && npm start` (Windows `cmd`) / `$env:PORT=3001; npm start`
  (PowerShell). Update the TV URLs to match.
- **A TV cannot reach the dashboard** — confirm the host IP (`ipconfig` / banner), that the
  TV is on the same subnet, and that the Windows Firewall Node.js rule allows Private
  networks. Try opening `http://<host-ip>:3000/health` from the TV's browser.
- **Windows Firewall blocked Node.js** — re-allow it under *Windows Defender Firewall →
  Allow an app through firewall* (Private).
- **Server PC IP changed** — set a DHCP reservation or static IP (see *Stable IP*); update
  the TV URLs.
- **Three.js / Chart.js didn't appear** — the CDN/internet was unavailable. The dashboard
  still runs; the occupancy ring shows its CSS fallback and charts simply don't draw.
  Restore internet (or, later, self-host the libraries).
- **Opened with `file://` instead of HTTP** — the standalone file works, but tenant-logo
  medallions fall back to initials (a `file://` security limit). Use `http://…:3000` for

- **Wrong/blank project** — check the `?project=` slug (`business-address` /
  `town-center`); an invalid value safely falls back to the default project.
- **Missing logo asset** — ensure `logos/al-tamimi.png` and `logos/tharwah_logo.png` exist;
  medallions fall back to initials if a logo is missing.
- **Node.js not installed / `npm` command not found** — install Node.js 20+ from
  nodejs.org (which provides `npm`), reopen the terminal, and retry.

## Adaptive quality (Phase 8.5)

The same URL adapts visual cost to the device. Append `?quality=`:

- `?quality=auto` (default) — detect + benchmark, pick full/reduced/static.
- `?quality=full` — full visuals (particle-wave, 3D occupancy, WebGL medallions).
- `?quality=reduced` — lighter: fewer particles, lower pixel ratio, flat DOM medallions, no panel float.
- `?quality=static` — no continuous WebGL (CSS background + CSS occupancy ring); best for old Smart TVs.

All live data, cards, KPIs, charts, tenant directory, building occupancy, search and 5-minute
polling are identical in every mode — only visual effects change. Combine with a project, e.g.
`http://<host>:3000/?project=town-center&quality=static`.

Reset the remembered auto result: `localStorage.removeItem('dashboard-quality-v1')` in the
browser console (or open `?quality=auto`). Remove a manual override by deleting `&quality=…`.
Developer overlay (localhost only): add `&qualityDebug=1`.
